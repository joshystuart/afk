**Function definition:**

```bash
setup_skills() {
    if [ ! -d "/home/afk/.skills" ]; then
        log_info "No skills directory mounted, skipping skills setup"
        return 0
    fi

    log_step "Setting up skills symlinks"

    # ~/.claude/skills/ — inside named volume (D-05), parent exists from volume mount
    # Remove any existing directory/symlink first, then create symlink
    rm -rf /home/afk/.claude/skills 2>/dev/null || true
    ln -sfn /home/afk/.skills /home/afk/.claude/skills

    # ~/.cursor/skills/ — parent may not exist in base image
    mkdir -p /home/afk/.cursor
    rm -rf /home/afk/.cursor/skills 2>/dev/null || true
    ln -sfn /home/afk/.skills /home/afk/.cursor/skills

    # ~/.agents/skills/ — parent may not exist in base image
    mkdir -p /home/afk/.agents
    rm -rf /home/afk/.agents/skills 2>/dev/null || true
    ln -sfn /home/afk/.skills /home/afk/.agents/skills

    # ~/.codex/skills/ — parent may not exist in base image
    mkdir -p /home/afk/.codex
    rm -rf /home/afk/.codex/skills 2>/dev/null || true
    ln -sfn /home/afk/.skills /home/afk/.codex/skills

    log_info "Skills symlinks created for all agent discovery paths"
}
```

Per D-02: symlinks to all 4 paths. Per D-03: at entrypoint runtime, not image build. Per D-04: conditional on mount existence (`[ ! -d "/home/afk/.skills" ]` check). Per D-05: `~/.claude/skills/` created inside the named volume at runtime.

The `rm -rf` before each `ln -sfn` ensures idempotency on container restarts (assumption A2 from research). The `mkdir -p` creates parent directories that don't exist in the base image (pitfall #2 from research).

<threat_model>

## Trust Boundaries

| Boundary                         | Description                                                                      |
| -------------------------------- | -------------------------------------------------------------------------------- |
| Host filesystem → Container      | Bind mount from user's host skills directory into container at /home/afk/.skills |
| Container process → Skills mount | Read-only enforcement via Docker `:ro` flag                                      |
| Entrypoint → Named volume        | Symlink creation inside Docker named volume at /home/afk/.claude                 |

## STRIDE Threat Register

| Threat ID | Category               | Component                          | Disposition | Mitigation Plan                                                                                                                                                                     |
| --------- | ---------------------- | ---------------------------------- | ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-02-02   | Elevation of Privilege | DockerContainerProvisioningService | mitigate    | `:ro` bind mount flag on skills mount — Docker engine enforces at kernel level. Code: `${options.skillsPath}:/home/afk/.skills:ro`                                                  |
| T-02-03   | Tampering              | entrypoint.sh                      | mitigate    | Symlinks point TO the read-only mount, not FROM it. `ln -sfn` creates links in writable locations; skills content itself cannot be modified                                         |
| T-02-05   | Denial of Service      | CreateSessionRequestService        | mitigate    | Skills directory validated with `MountPathValidator.validate()` at session creation. Invalid/missing paths are rejected with a clear error before Docker API call                   |
| T-02-06   | Information Disclosure | Container via skills mount         | accept      | Skills are user-provided content intentionally shared with the container. The user controls what's in the skills directory. Read-only prevents data exfiltration back to the mount. |

</threat_model>
